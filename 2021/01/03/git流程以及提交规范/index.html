Git工作流程和commit提交规范

Git工作流程和分支规范

gitflow分支管理方式是现在非常流行的一套git工作流程具体创建以下几种分支：

  • feature branches。功能分支，一般是从 develop 开发分支上检出(checkout)
  • develop。公共的开发主线分支，feature 功能分支的代码开发完成后，经过 code review 后会合并到此分支
  • release branches。测试、发布主线，此分支是从 develop 分支上检出(checkout), 一般是提测阶段会使用该分支的代码
  • bugfix。修复 release 分支问题，
  • hotfix。紧急修复，一般是用于修复上线后的生产环境的问题。
  • master。可发布的稳定版分支

功能开发流程

一般开发人员拿到需求之后从develop分支签出一个功能分支，命名可以以feat-xxx，然后在此分支上进行开发，开发完成后提交到远程分支，并在对应的远程仓库管理软件中提交merge request，经过相关人员的code view 之后将功能分支合并到develop分支上。

提测流程

当开发人员完成一个版本的所有任务之后，向测试人员提出版本测试请求。此时develop分支上会包含该版本所有新需求的代码。测试人员从develop分支上检出release分支，命名格式为release-xxx，QA人员会基于此分支进行代码测试以及测试环境部署。当QA提出bug的时候，开发人员从release-xxx分支签出bugfix-xx分支修复后合并到release-xxx分支上。

部署发布流程

当测试人员测试通过后，会将release-xx分支合并到master分支上，并将release-xx分支合并到develop分支，将master 分支进行部署并测试线上环境。正式发布时运维人员将master分支部署到线上环境打上tag，并合并一次develop分支。这里tag相当于一个commit方便线上环境出现重大问题时能准确回滚到想要的版本。

线上紧急问题修复

当线上环境出现bug时，需要在master分支上签出一个hotfix-xxx分支，然后开发人员修复代码后合并到hotfix-xxx分支上，通过测试后将其合并到master与develop分支。然后在master分支上打上tag。

Git提交规范

在多人协同开发时制定一套统一的git commit信息提交规范时非常有必要。它能让我们在查找和定位代码问题时节省精力，同时打印git log 时也更清晰。下面是我们小团队参照angularjs团队的commit制定的一套commit信息规范。

commit信息包含三个部分：type(必需)，scope(可选)，subject(必需)。
  1. type: type用于说明本次commit的类型，一般有以下几种：
    • feat: 新功能(feature)
    • fix: 修复 bug
    • docs: 文档(documents)
    • perf：优化
    • style: 代码格式(不影响代码运行的格式变动，注意不是指 CSS 的修改)
    • refactor: 重构(既不是新增功能，也不是修改 bug 的代码变动)
    • test: 提交测试代码(单元测试，集成测试等)
    • chore: 构建或辅助工具的变动
  2. scope： 一般scope用来说明本次提交影响的功能模块
  3. subject：用与对commit的简短描述
  4. ID: 这里的ID一般是公司项目管理工具中对应case的id 多用于bug修复时记录。
