基于vue-cli 3.0实现一个工程多项目多模块打包

Mar 1, 2020 | webpack

整合多个项目时探究对于一个工程多项目独立打包的实现

前言

最近公司在整合多个CMS的项目，想实现根据配置文件只给予toB客户对应的模块功能。这样的需求对于toB的大型CMS项目来说是比较常见，而且这次我们想做的是从代码层面就不给予客服非付费模块。

因此在不对业务代码过多修改的前提下利用webpack打包机制直接按配置打包所需模块是目前想到的最优解。

webpack打包原理

Webpackl根据入口(entry) 提供的入口文件开始分析其直接或者间接的依赖模块或者库来构建依赖图打包。在vue-cli的脚手架中，以mian.js为默认打包入口遍历其中引入的依赖包和路由中引入的组件以及其组件依赖。根据webpack的打包机制我们就可以有两种思路来实现一个工程多模块按需打包，1构建多个入口，2共用一个入口构建不同的依赖

1、多入口

根据所需模块构建文件夹以及入口文件(main.js), 编译时根据环境变量配置使用对应的入口文件。

项目目录

我们还需要创建一个multiConfig 配置来定义每个独立模块的端口入口文件地址以及有需要时定义api本地代理地址。

因为项目为vue-cli 3.0搭起来的所有我们需要在vue.config.js中更改相应配置

为了方便直接启动不同模块我们需要在package.json内的scripts中添加对应的script

到这里我们就可以直接使用npm run serve:project1 来启动模块1的项目。

优点：1不同模块使用的依赖包可按需打包，2各模块静态资源可独立配置，3模块之间相互独立编译、打包并且可以直接引用使用。

缺点：每个需要单独构建的模块都需要建立main.js,ruoute等入口及配置文件，条件过于繁琐。

2、共用入口文件利用依赖不同构建

所有模块共同一套依赖库，根据不同的路由配置文件引入的依赖组件不同打包所需要的模块。

项目目录

对于如何动态引入路由配置文件时，我们可以利用NormalModuleReplacementPluginwepack内置插件，该插件允许替换引入资源匹配的关键词。

在package.json中我们对scripts新增对应启动命令来启动或打包对应模块"serve:project1": "vue-cli-service serve --mode project1", "build:project1": "vue-cli-service serve --mode project1"

优点：1除开公共模块外根据router配置按需打包，2改动小，只需配置对应的router文件

缺点：1对于依赖包无法做到按需加载，2因为共用同一套webpack配置没法做到同时启动多个项目

总结

两种方法各有千秋但都是根据引入依赖不同来构建所需的模块。对于模块ui风格统一、依赖包区别不大的工程利用第二种方法来改造更合适毕竟只需要配置所需的模块路由文件就可以了。
